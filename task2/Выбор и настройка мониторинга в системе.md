# Выбор и настройка мониторинга в системе

## Мотивация

* Внедрение мониторинга позволяет своевременно находить узкие места в архитектуре и оперативно устранять проблемы до того, как они отразятся на пользователях;  
* Повышение стабильности работы сервисов напрямую улучшает качество клиентского опыта;  
* Наличие данных о загрузке компонентов даёт возможность гибко управлять инфраструктурой — перераспределять ресурсы и оптимизировать расходы на оборудование и поддержку.  

## Подход к мониторингу

* Для MES целесообразно применять методологию **Four Golden Signals** или **USE**, так как они позволяют детально отслеживать как производительность, так и состояние ресурсов;  
* Для `CRM` и `Интернет-магазина` подходит **RED-подход**, ориентированный на метрики доступности и качества работы API;  
* Для баз данных и системы обмена сообщениями наиболее эффективен **USE-подход**, так как он сфокусирован на ресурсах, насыщении и ошибках.  

## Метрики

### MES
* Объём хранилища S3  
* Количество запросов (RPS) к MES API — показатель нагрузки  
* RPS в разрезе пользователей — помогает выявить аномальное использование  
* Количество HTTP 200 — успешные ответы  
* Количество HTTP 500 — ошибки системы  
* Одновременные активные сессии — текущая нагрузка на сервис  

### CRM
* Количество запросов (RPS) к CRM API  
* RPS по пользователям  
* Количество успешных HTTP 200 ответов  
* Количество HTTP 500 ошибок  
* Число параллельных сессий  

### Интернет-магазин
* Количество запросов (RPS) к API магазина  
* Запросы в разрезе пользователей (RPS)  
* Использование памяти в экземпляре базы магазина — индикатор насыщения  
* Количество HTTP 200  
* Количество HTTP 500 — ошибки работы API  
* Одновременные активные сессии пользователей  

### Очередь сообщений
* Число сообщений в dead-letter-очереди RabbitMQ — индикатор ошибок обработки  
* Количество сообщений «в полёте» в RabbitMQ — текущая нагрузка системы обмена  

### PostgreSQL
* Использование памяти на БД MES  
* Использование памяти на БД магазина  
* Количество подключений к БД магазина — может указывать на утечки соединений или перегрузку  
* Количество подключений к БД MES  
* Размер БД магазина — помогает заранее планировать масштабирование или шардирование  
* Размер БД MES  

## План внедрения

* Развернуть и настроить систему сбора метрик **Prometheus**;  
* Организовать экспорт метрик из сервисов;  
* Установить и сконфигурировать **Grafana**, подключив её к Prometheus;  
* Подготовить дашборды для визуализации ключевых показателей;  
* Настроить систему оповещений в Grafana для автоматического реагирования на инциденты.
