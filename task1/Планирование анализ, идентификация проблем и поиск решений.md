# Планирование: анализ, идентификация проблем и поиск решений
## Проблемные места в системе
- Ручное тестирование без автоматизации, что замедляет процессы и повышает риск ошибок.
- Отсутствие кэширования, из-за чего возрастает нагрузка на базу данных.
- Сервисы не масштабируются под растущую нагрузку.
- База данных работает без репликации и шардирования, что создаёт риски потери данных и простоев.
- Высокий bus factor - по одному специалисту на ключевые роли: C#, React, тестирование, DevOps.
- Развёртывание в dev и release окружениях выполняется вручную.
- Нет использования Kubernetes и Feature Toggles для управления релизами.
- Процессы релиза замедлены и нестабильны.
- Ограниченное логгирование, отсутствует распределённый трейсинг и централизованный мониторинг.
- RabbitMQ работает без репликации, что делает его уязвимым к сбоям.

## Список инициатив
1. Внедрение системы наблюдаемости: логгирование, мониторинг, трейсинг. Настройка ELK, Prometheus и Grafana для сбора 
   и анализа данных о работе системы.
2. Добавление кэширования с использованием Redis. Ускорение обработки запросов за счёт кэширования часто используемых 
   данных.
3. Настройка репликации базы данных. Создание реплик для чтения, повышение отказоустойчивости и производительности.
4. Найм новых сотрудников. Расширение команды по направлениям: C#, Vue, React, DevOps для снижения зависимости от 
   отдельных специалистов. 
5. Повышение квалификации команды. Подготовка к переходу на Kubernetes и микросервисную архитектуру.
6. Автоматизация тестирования. Внедрение unit, интеграционных и end-to-end тестов.
7. Настройка инструментов мониторинга (ELK, Prometheus, Grafana). Обеспечение единой точки контроля за состоянием 
   сервисов.
8. Миграция ключевых сервисов на Kubernetes.
9. Оптимизация CI/CD процессов. Автоматизация релизов, внедрение частичных обновлений и механизмов отката.

## Целевая архитектура

Целевая архитектура включает в себя:
- Внедрение кэширования через Redis для снижения нагрузки на БД.
- Наличие реплик базы данных для операций чтения и повышения отказоустойчивости.
- Реализованную систему наблюдаемости: логи, метрики, трейсы - в единой панели.
- Расширенную и обученную команду, готовую к работе с современной инфраструктурой.
- Автоматизированные процессы тестирования и доставки кода.
- Подготовку к переходу на оркестрацию с помощью Kubernetes.
- Повышенную стабильность и управляемость системы в целом.

## Приоритетные инициативы (топ-3)
- Внедрение наблюдаемости: Без понимания состояния системы невозможно оценить эффективность других изменений. Это 
  базовый элемент для диагностики и контроля.
- Добавление кэширования (Redis): Простое в реализации улучшение, которое сразу снижает нагрузку на базу данных и 
  ускоряет работу сервисов.
- Настройка репликации базы данных: Повышает надёжность хранения данных, устраняет одиночную точку отказа и улучшает 
  производительность при чтении.

Эти три инициативы решают наиболее критичные технические риски, не требуют масштабной перестройки системы и могут быть реализованы с текущей командой и ресурсами. Они создают основу для дальнейшего развития архитектуры.
